Why ADA ?

The engine of text processing is Regular expression based on UTF-32. There are other regular expression implementations in standard libraries of ADA but are based on a "character" (8-bit entity, ASCII). A character as € (euro) is in UTF-8 3*8 bits but in UTF-32 one 32 bit character.

Following features and goals gives Y2018.Text an edge in text processing
  1. Strong typing - makes it possible to process text stored as UTF-8 or UTF-32.
  2. Smooth transmission between different storage methods.
  3. No virtual machine level, direct access to the native platform
  4. Unicode support 
To achieve a base for these goals we needed a way to fix the literal problem, the '&'-operator problem, the Wide_Wide_String and -character problem and to solve to function-selector problem. With the Y2018Text preprocessor this is done, although the preprocessor is not in-line with ADA ideology.

[2] Smooth transmission between different character storage methods

It is important to make it possible to concatenate text of different storage methods. By using "+"-operator (as a stand-in for "&") and overloading conversions between UTF-8 and UTF-32 is (almost) invisible.

New Wide_Wide_String literal notation with hexadecimal values and some control characters (BS, HT, LF, CR and FF).

Unbounded_String and Unbounded_Wide_Wide_String can be concatenated directly with String and Wide_Wide_String with the "+"-operator.

[3] No virtual machine level, direct access to the native platform

No JVM or anything else like it.

[4] Unicode support

Most computer languages support only UTF-8 and UTF-16, but here the support is based on UTF-32. As a backup to this support is a regular expression implementation based on UTF-32. The support still lacks features as language based collation sequence, normalization of text and support for specific scripts. What is supported is upper-, lower- and tittle case simple transformation and general character classes. All Unicode code-points are supported at Unicode standard.
