Why ADA ?

The engine of text processing is Regular expression based on UTF-32. There are other regular expression implementations in standard libraries of ADA but are based on a "character" (8-bit entity, ASCII). A character as € (euro) is in UTF-8 3*8 bits but in UTF-32 one 32 bit character.

Best Regular expression support is in Perl script and in this implementation the Perl implementation is used as a model all tough some features can not be implemented in ADA or are too complicated to be implemented. Some features in our implementation has no corresponding feature in Perl because of UTF-32 or the ADA language.

Design Patterns for the use of our regular expression is presented in the documentation (Chapter 1 and 4.4.1). The biggest difference from Perl is that we do not have a way to specify a line and an "end of line". Meta-characters "^" and "$" are always "start-of-data" and "end-of-data" (same as Perl search pattern with modifier "s"). Patterns can have modifiers, but only Uppercase is here implemented.

Following features and goals gives Ada an edge in text processing

  1. Strong typing - makes it possible to process text stored as ASCII and EBCDIC on different code pages.
  2. Smooth transmission between different storage methods.
  3. No virtual machine level, direct access to the native platform
  4. Unicode support 
To achieve a base for these goals we needed a way to fix the literal problem, the '&'-operator problem, the Wide_Wide_String and -character problem and to solve to function-selector problem. With the Y2018Text preprocessor this is done, although the preprocessor is not in-line with ADA ideology.

[1] Strong typing - makes it possible to process text stored as ASCII and EBCDIC on different code pages

It is important that ASCII or EBCDIC text is not defined as Character or String. A specific enumeration type and corresponding array type must be used. Character and String are for UTF-8 only (Wide_Wide_Character and Wide_Wide_String for UTF-32).
ASCII and EBCDIC are both defined for 128/256 characters and for both exits a hugh amount of different "code pages". Unicode has only one code page, but is defined for 2**21 code points and with five different storage methods (UTF-8, UTF-16BE, UTF-16LE, UTF-32BE, UTF-32LE).

[2] Smooth transmission between different character storage methods

It is important to make it possible to concatenate text of different storage methods. By using "+"-operator (as a stand-in for "&") and overloading conversions between UTF-8 and UTF-32 is (almost) invisible.

New Wide_Wide_String literal notation with hexadecimal values and some control characters (BS, HT, LF, CR and FF).

Unbounded_String and Unbounded_Wide_Wide_String can be concatenated directly with String and Wide_Wide_String with the "+"-operator (see examples and Sample in Y2018-Text project).

[3] No virtual machine level, direct access to the native platform

No JVM or anything else like it.

[4] Unicode support

Most computer languages support only UTF-8 and UTF-16, but here the support is based on UTF-32. As a backup to this support is a regular expression implementation based on UTF-32. The support still lacks features as language based collation sequence, normalization of text and support for specific scripts. What is supported is upper-, lower- and tittle case transformation and general character classes. All Unicode code-points are supported at Unicode standard version 12.
